## 競合調査・差別化

### 主要競合サービスの課題

| サービス | 課題・弱点 |
|---|---|
| **RoomSketcher** | 有料プランが必須・操作が複雑・英語UI |
| **Planner 5D** | 3D特化で間取り図出力に向かない・アカウント登録必須 |
| **madori** | 日本語対応だがモバイル向け・PCでの精密編集が弱い |
| **floorplanner.com** | 無料版は出力に透かし・サーバー依存 |

### このツールの差別化ポイント

1. **手描きからのAI変換**：スキャン/写真を撮るだけでデジタル化 → 現場スタッフの作業時間を大幅削減
2. **ゼロセットアップ**：URLを開くだけ、インストール・アカウント不要
3. **社内特化**：不動産業務に絞ったシンプルUI（3D不要、余計な機能なし）
4. **完全無料**：GitHub Pages + Gemini API無料枠
5. **オフライン対応可能**：手動作成機能はAPIなしで動作

## 技術スタック

### 採用構成

```
フロントエンド: Vanilla HTML/CSS/JavaScript（フレームワークなし）
キャンバス描画: Fabric.js v6
スタイリング: CSS変数ベースのカスタムデザイン（Tailwindなし）
AI連携: Google Gemini API（gemini-2.0-flash）
ホスティング: GitHub Pages
出力: Canvas API → PNG / SVG.js → SVG
```

### 技術選定の理由

**Fabric.js を採用する理由**
- ドラッグ&ドロップ、スナップグリッド、オブジェクト選択・変形がすべて標準装備
- Canvas/SVG 両対応 → PNG/SVG 出力が容易
- 学習コストが低く、将来の保守担当者でも読みやすい

**フレームワークなし（Vanilla JS）の理由**
- GitHub Pages での静的配信と相性が良く、ビルドプロセスが不要
- ファイル構成がシンプルで非エンジニアでもフォルダを開けば理解できる
- React/Vue はオーバーキル（SPA不要）

**Gemini APIを直接呼ぶ理由**
- サーバーレス = 維持費ゼロ
- `gemini-2.0-flash` はマルチモーダル対応で間取り図解析に十分

## 実装フェーズ計画

### フェーズ1：プロジェクト基盤 + エディター骨格
- HTML/CSS の全体レイアウト（3カラム）
- Fabric.js のキャンバス初期化
- グリッド表示・スナップ機能
- **確認ポイント**: ブラウザで開いてグリッド付きキャンバスが表示される

### フェーズ2：部屋・壁の描画機能
- 矩形（部屋）の配置・リサイズ・移動
- ラベル入力（部屋名・畳数）
- 壁の線描画ツール
- Undo/Redo（Ctrl+Z/Y）
- **確認ポイント**: 部屋を複数置いて名前をつけ、動かせる

### フェーズ3：ドア・窓・その他パーツ
- ドア（開き方向付き弧）・窓の配置
- 階段・テキストラベル
- プロパティパネル（選択中オブジェクトの編集）
- **確認ポイント**: ドア・窓を壁に配置して編集できる

### フェーズ4：出力機能
- PNG エクスポート（解像度指定）
- SVG エクスポート
- ローカルへの JSON 保存・読み込み（作業途中の保存）
- **確認ポイント**: PNG/SVGが正常なファイルとしてダウンロードできる

### フェーズ5：AI手描き読み込み機能
- 画像アップロードUI
- Gemini API 呼び出し（プロンプト設計）
- レスポンスJSONをFabric.jsオブジェクトに変換
- 解析結果のプレビュー → エディターへ引き渡し
- **確認ポイント**: 手描き写真をアップロードして間取り図が生成される

### フェーズ6：品質仕上げ（任意）
- モバイル対応
- スケール表示・実寸入力
- 寸法線の自動表示

## リスク・課題

| リスク | 影響度 | 対策 |
|---|---|---|
| Gemini APIの解析精度が低い | 高 | プロンプトの最適化・ユーザーへの補正UIを充実させる |
| APIキーの露出 | 高 | 後述のセキュリティ対策を参照 |
| Fabric.js の学習コスト | 中 | ドキュメントが充実・サンプル多数 |
| 大きな間取り図でのパフォーマンス | 低 | オブジェクト数が多い場合のみ最適化を検討 |
| 異なるブラウザでの表示差異 | 低 | Chrome/Edge/Safari で都度確認 |

## セキュリティ対策案

### 現状リスク（APIキーをコードに埋め込む場合）

- GitHubにpushするとキーが公開される
- Google がボットスキャンでキーを検出し自動無効化することもある
- 第三者が盗用して API 使用量を消費される

### 対策案

**案A：Gemini APIの「API制限」機能を使う（推奨・初期実装向け）**
- Google Cloud Console でキーの「HTTPリファラー制限」を設定
- `https://[GitHubPages URL]/*` のみ許可
- コストゼロ・作業5分

**案B：Cloudflare Workers を Proxy にする（中期対応）**
- 無料枠：10万リクエスト/日
- Workers にキーを環境変数として保存 → フロントからはWorkers URLにリクエスト
- キーはコードに出てこない
- セットアップ：30分程度

**案C：GitHub Secrets + GitHub Actions でビルド時注入（上級）**
- CI/CDパイプラインでキーを環境変数として埋め込んでビルド
- 開発コストが高いため後回し推奨

**初期実装の推奨方針**：案Aを即時実施 + 案Bを追加フェーズとして計画
